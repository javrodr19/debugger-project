# Aegis Debug V1.1 — Syntax & Compilation Error Detection

**Date:** 2026-04-15
**Status:** Design approved, ready for implementation planning
**Version target:** `1.1.0`
**Base branch:** `main`

---

## 0. Summary

V1 ships with five static analyzers aimed at high-level logic patterns (null safety, state-before-init, async flow, circular dependencies, complexity). None of them catch what the user considers the most basic class of "broken code": syntax and compilation errors. A file with `val classment = ` slips through Aegis Debug unflagged even though IntelliJ's own compiler underlines it in red.

V1.1 closes this gap. Two new deterministic early-pass analyzers catch parse-level syntax errors and IDE-reported compilation errors, respectively. When a file has either class of error, downstream analyzers skip it to prevent false-positive cascades on broken ASTs.

This document is the approved design. An implementation plan will follow via the `writing-plans` skill.

---

## 1. Scope decisions (locked in)

| # | Decision | Rationale |
|---|---|---|
| Q1 | **Detect both PSI parse errors and IDE daemon ERROR-severity highlights** | Covers `PsiErrorElement` (parse failures) *and* semantic errors the IDE reports (unresolved references, type mismatches). "Syntax and general compilation errors" per user's framing. |
| Q2 | **Skip downstream analyzers on broken files** | Prevents false-positive cascades (e.g., `NullSafetyAnalyzer` firing on variables the parser couldn't understand). Matches the "trust is the product" principle from the V1 spec. |
| Q3 | **Full sweep across the analysis set** — no separate cap, no "files of interest" prioritization | Senior-engineer-auditing-a-codebase use case wants complete coverage. Accepted runtime cost: ~10–30 s added on medium repos. |
| Q4 | **ERROR severity only** | Literal mapping of "syntax and compilation errors". WARNING-severity harvest deferred to V1.2 as a settings toggle. |

### Non-goals

- No deterministic fixer for syntax/compilation errors (can't be guaranteed safe).
- No WARNING-severity harvest.
- No separate `maxFilesForSyntaxCheck` cap.
- No "changed files only" prioritization.
- No language support beyond the existing `FileScanner.SUPPORTED_EXTENSIONS` set (`kt`, `java`, `ts`, `tsx`, `js`, `jsx`).

### 2.1 Package layout

```

│     ├─ CompilationErrorAnalyzer.kt   (new — AEG-COMPILE-001)
│     ├─ NullSafetyAnalyzer.kt         (unchanged)
│     ├─ StateInitAnalyzer.kt          (unchanged)
│     ├─ AsyncFlowAnalyzer.kt          (unchanged)
│     ├─ CircularDependencyAnalyzer.kt (unchanged)
│     └─ ComplexityAnalyzer.kt         (unchanged)
└─ model/
   └─ AnalysisModels.kt                (modified — add two IssueType values)
```

### 2.2 Analyzer hierarchy

`Analyzer.kt` gains an empty marker interface:

```kotlin
/** Marker for analyzers that must run before all others so the engine can
 *  compute the set of broken files and skip downstream analyzers on them. */
interface EarlyAnalyzer : Analyzer
```

Both new analyzers implement `EarlyAnalyzer`. Existing five implement plain `Analyzer`. No behavior change to existing classes.

### 2.3 Approach chosen

**Approach 1 — Two dedicated analyzers + early-pass engine hook.** Rejected alternatives: single combined analyzer (loses per-rule attribution), preprocessing phase outside the `Analyzer` interface (breaks uniformity).

## 3. Data model

Two new values in `IssueType`:

```kotlin
@Serializable
enum class IssueType {
    SYNTAX_ERROR,          // new — PSI parse failure
    COMPILATION_ERROR,     // new — IDE daemon ERROR-severity highlight
    NULL_SAFETY,
    CIRCULAR_DEPENDENCY,
    UNHANDLED_PROMISE,
    STATE_BEFORE_INIT,
    HIGH_COMPLEXITY,
    MISSING_ERROR_HANDLING,
    DEAD_CODE,
    RESOURCE_LEAK,
    MEMORY_LEAK,
    ARCHITECTURE
}
```

No `Issue` / `CodeFix` / `ProjectMetrics` shape changes — existing fields (`ruleId`, `line`, `column`, `codeSnippet`, `severity`, `sources`, `providers`) carry everything needed.

### Rule IDs (following existing `AEG-<CATEGORY>-<NNN>` convention)

| Analyzer | Rule ID | Issue type | Severity |
|---|---|---|---|
| `PsiSyntaxAnalyzer` | `AEG-SYNTAX-001` | `SYNTAX_ERROR` | `ERROR` |
| `CompilationErrorAnalyzer` | `AEG-COMPILE-001` | `COMPILATION_ERROR` | `ERROR` |

## 4. Engine flow

`AnalysisEngine.analyze()` is restructured into two static phases:

```
1. Scan + parse + graph build                   (unchanged)
2. Early static pass
   - Partition registered analyzers:
     early = analyzers.filterIsInstance<EarlyAnalyzer>()
     late  = analyzers.filterNot { it is EarlyAnalyzer }
   - Run `early` concurrently (Dispatchers.Default, one async per analyzer)
   - All files in context are eligible
3. Compute brokenFilePaths
   - Set of normalized paths where ANY early analyzer emitted an issue
   - Normalized = path.replace("\\", "/")
4. Build filtered context for downstream passes
   - filteredContext = context.copy(
       parsedFiles = context.parsedFiles.filterNot { it.path.replace("\\", "/") in brokenFilePaths }
     )
   - Graph stays full — broken files still appear in the NeuroMap with ERROR status
   - filesByPath lazy map on the filtered ParsedFile list regenerates automatically
5. Late static pass (existing five analyzers)
   - Runs on filteredContext — broken files are not visible to them
6. AI pass (Ollama / OpenAI)
   - Runs on filteredContext — no point asking an LLM about a file that won't parse
   - **Behavior change:** V1 called `runAiPass(limitedContext, …)`. V1.1 calls `runAiPass(filteredContext, …)` where `filteredContext` is `limitedContext` further filtered to exclude broken files. The `maxAiFiles` cap still applies on top of the filtered set.
7. Merge issues (early + late + AI)
   - Existing fingerprint-based dedupe (ruleId:filePath:line) handles all cases
8. Metrics, hotspots, risks
   - Syntax and compilation issues count as ERROR severity (existing formula)
```

### De-duplication between PSI-error and IDE-daemon overlap

A parse failure is often reported by *both* analyzers (the daemon includes parser errors in its highlight stream). They produce different `ruleId`s (`AEG-SYNTAX-001` vs `AEG-COMPILE-001`), so `Issue.fingerprint()` treats them as distinct. This is **intentional** — showing "both pipelines flagged this" is a stronger trust signal, and users can see which detection path caught what. If empirical data shows this is noisy, a post-merge step preferring `AEG-COMPILE-001` on identical `(filePath, line)` pairs can be added in V1.2.

## 5. Component: `PsiSyntaxAnalyzer`

### 5.1 Contract

```kotlin
class PsiSyntaxAnalyzer : EarlyAnalyzer {
    override val name = "PsiSyntaxAnalyzer"
    override val ruleId = "AEG-SYNTAX-001"
    override val defaultSeverity = IssueSeverity.ERROR
    override val description = "Detects parse-level syntax errors across supported languages using the IDE's PSI tree."
    override fun analyze(context: AnalysisContext): List<Issue>
}
```

### 5.2 Detection logic

For each `ParsedFile`:

1. Wrap in `ApplicationManager.getApplication().runReadAction { … }`.
2. Resolve PSI: `PsiManager.getInstance(context.project).findFile(parsedFile.virtualFile)`. Skip if null (unknown language, plugin not loaded).
3. Walk: `PsiTreeUtil.collectElementsOfType(psiFile, PsiErrorElement::class.java)`.
4. For each `PsiErrorElement`:
   - Resolve document: `PsiDocumentManager.getInstance(project).getDocument(psiFile)`; skip if null.
   - `offset = element.textRange.startOffset`
   - `line = document.getLineNumber(offset) + 1` (1-indexed, matches existing convention)
   - `column = offset - document.getLineStartOffset(line - 1) + 1`
   - `title = "Syntax error: ${element.errorDescription}"` (IntelliJ's own message; examples: "Expecting ';'", "Unexpected token")
   - `description = "${element.errorDescription}. Aegis Debug will skip additional analysis on this file until the parse error is resolved."`
   - `codeSnippet` = `parsedFile.lines` sublist `[maxOf(0, line-3), minOf(lines.size, line+2)]` joined by `\n` (matches the 5-line window used by `NullSafetyAnalyzer`)
5. Emit an `Issue` per error with the above fields and the standard provenance tagging (`sources = [STATIC]`, `providers = [STATIC]`, `confidence = 1.0`) — applied by the engine's existing `runOne` wrapper, no change needed.

### 5.3 Cancellation

Call `ProgressManager.checkCanceled()` before each file and inside the inner error iteration (files with dozens of errors exist). Matches the pattern already used by `GhostDebuggerAnnotator.doAnnotate`.

### 5.4 Edge cases

| Case | Behavior |
|---|---|
| Empty file | No `PsiErrorElement` collected → emit nothing. |
| File parses cleanly, has semantic errors only | `PsiSyntaxAnalyzer` emits nothing; `CompilationErrorAnalyzer` catches them. |
| `findFile` returns null (binary, unknown language) | Skip silently — no emit, no log. |
| `document` is null (e.g., large file lazy-loaded) | Skip the file, log once with `log.warn`. |
| Line out of `parsedFile.lines` bounds | `coerceIn(0, lines.size)` — same guard as `NullSafetyAnalyzer`. |

### 5.5 Threading

Fully background-safe. All PSI access wrapped in `ReadAction`. Parallelizes via the existing `analyzers.map { async(Dispatchers.Default) { … } }` fan-out in `runEarlyStaticPass`.

### 5.6 Language coverage

Any extension in `FileScanner.SUPPORTED_EXTENSIONS` that has a PSI parser loaded in the current IDE. Kotlin and Java always work (native). TypeScript/JavaScript depend on the JavaScript plugin — bundled in Ultimate, separate install in Community. If unavailable, `findFile` returns null and the analyzer emits nothing for that file. `CompilationErrorAnalyzer` provides a second detection path for the same files.

## 6. Component: `CompilationErrorAnalyzer`

### 6.1 Contract

```kotlin
class CompilationErrorAnalyzer : EarlyAnalyzer {
    override val name = "CompilationErrorAnalyzer"
    override val ruleId = "AEG-COMPILE-001"
    override val defaultSeverity = IssueSeverity.ERROR
    override val description = "Surfaces IDE-reported compilation errors (unresolved references, type mismatches, invalid declarations) harvested from the IntelliJ analysis daemon."
    override fun analyze(context: AnalysisContext): List<Issue>
}
```

### 6.2 Detection pipeline (per file, concurrency-bounded)

1. `ApplicationManager.getApplication().runReadAction { … }`.
2. Resolve PSI: `PsiManager.getInstance(project).findFile(virtualFile)`; skip if null.
3. Resolve document: `FileDocumentManager.getInstance().getDocument(virtualFile)`; skip if null.
4. Invoke the daemon synchronously:
   ```kotlin
   DaemonCodeAnalyzerImpl
       .getInstanceEx(project)
       .runMainPasses(psiFile, document, EmptyProgressIndicator())
   ```
   Returns `List<HighlightInfo>`.
5. Filter to `severity == HighlightSeverity.ERROR`.
6. For each surviving highlight, emit an `Issue` with:
   - `line = document.getLineNumber(highlight.startOffset) + 1`
   - `column = highlight.startOffset - document.getLineStartOffset(line - 1) + 1`
   - `title = "Compilation error: ${highlight.description?.take(120) ?: "unspecified"}"`
   - `description = highlight.toolTip?.let(::stripHtml) ?: highlight.description ?: "IDE reported an error at this location."`
   - `codeSnippet` = same 5-line window as `PsiSyntaxAnalyzer`
   - Provenance: `sources = [STATIC]`, `providers = [STATIC]`, `confidence = 1.0` (applied by engine's `runOne` wrapper)

### 6.3 Concurrency control

`runMainPasses` is CPU-heavy — it runs all registered highlighting passes (annotators, inspections, reference resolution). Unbounded fan-out on 500 files thrashes. Bounded coroutine pattern (mirrors the Ollama concurrency control in `AnalysisEngine.runOllamaPass`):

```kotlin
private val semaphore = Semaphore(DAEMON_CONCURRENCY)  // 4

override fun analyze(context: AnalysisContext): List<Issue> = runBlocking {
    coroutineScope {
        context.parsedFiles.map { file ->
            async(Dispatchers.Default) {
                semaphore.withPermit {
                    ProgressManager.checkCanceled()
                    harvestFile(file, context.project)
                }
            }
        }.awaitAll().flatten()
    }
}
```

Note: `runBlocking` inside the analyzer is acceptable because the engine already calls each analyzer from an `async(Dispatchers.Default)` block, so we're nested coroutines, not blocking an EDT thread. Alternative: make `Analyzer.analyze` a `suspend fun` — rejected as a cross-cutting interface change for V1.1; revisit if more analyzers need concurrency.

### 6.4 Reaching past the public facade — trade-off

`DaemonCodeAnalyzerImpl` is `@ApiStatus.Internal`. Alternatives considered:

| Alternative | Why rejected |
|---|---|
| `CodeInsightTestFixture.doHighlighting()` | Test-framework only, not runtime-safe. |
| `HighlightInfoFilter` + wait for daemon | Fully async, timing-dependent, fragile across IDE refreshes. |
| `DaemonCodeAnalyzerEx.getFileLevelHighlights()` | File-level only; misses range highlights (the majority). |

**Decision:** use `DaemonCodeAnalyzerImpl.runMainPasses` with defensive wrapping:

```kotlin
try {
    // … runMainPasses call …
} catch (t: Throwable) {
    // NoSuchMethodError / LinkageError / anything else
    log.warn("daemon harvest failed for ${file.path}: ${t.message}")
    emptyList()
}
```

Verified against the full `pluginVerification` matrix in `build.gradle.kts` (2023.2.6 → 2025.1). If a future IDE breaks this API, the analyzer degrades gracefully (PSI path still fires) and logs provide the trigger for a hotfix.

### 6.5 HTML stripping for `toolTip`

`HighlightInfo.toolTip` is rich-text HTML (rendered by the Problems panel). Our `description` field is plain text (rendered in the tool window and Problems tooltip). Strip tags with a minimal regex:

```kotlin
private fun stripHtml(s: String): String = s.replace(Regex("<[^>]+>"), "").trim()
```

No HTML parser dependency added.

### 6.6 Performance budget

| Project size | Per-file cost (avg) | Concurrency | Added runtime |
|---|---|---|---|
| 100 files | ~80 ms | 4 | ~2 s |
| 250 files | ~100 ms | 4 | ~6 s |
| 500 files | ~120 ms | 4 | ~15 s |

Cost scales roughly linearly with file size and inspection-profile complexity. Documented in the changelog under "Changed" so users know what they're paying for.

### 6.7 Fallback behavior (all paths emit empty list; never fail the whole analysis)

| Failure | Behavior |
|---|---|
| `findFile` returns null | Skip, no log. |
| `getDocument` returns null | Skip, log once per file. |
| `runMainPasses` throws | Skip, log once per file with exception message. |
| Whole-analyzer unhandled exception | Caught by engine's existing `runOne` wrapper; logged; analysis continues. |

## 8. Testing strategy

### 8.1 Unit (JVM-only, existing pattern)

- **`AnalyzerContractTest`** (existing, extended): add `PsiSyntaxAnalyzer()` and `CompilationErrorAnalyzer()` to the list. Verifies non-blank metadata, unique rule IDs, `AEG-<CATEGORY>-<NNN>` pattern match, valid severity enum.

### 8.2 Integration (IntelliJ Platform test framework — `BasePlatformTestCase`)

Two new files under `src/test/kotlin/com/ghostdebugger/analysis/analyzers/`:

**`PsiSyntaxAnalyzerTest`** (extends `BasePlatformTestCase`):
- Kotlin `val classment = ` (unterminated) → one `AEG-SYNTAX-001`, ERROR severity, correct line.
- Kotlin `fun foo() {` (unclosed brace) → at least one `AEG-SYNTAX-001`.
- Java `public class { }` (missing class name) → one `AEG-SYNTAX-001`.
- Clean Kotlin `fun foo() = 42` → empty list.
- Binary/unreadable file path → empty list, no exception.

**`CompilationErrorAnalyzerTest`** (extends `BasePlatformTestCase`):
- Kotlin `val x: Int = "string"` (type mismatch) → one `AEG-COMPILE-001`.
- Kotlin referencing undefined symbol `val x = doesNotExist()` → one `AEG-COMPILE-001`.
- Clean Kotlin file → empty list.
- File with parse error AND semantic error on different lines → one `AEG-SYNTAX-001` and one `AEG-COMPILE-001`, both emitted, different fingerprints.

### 8.3 Engine integration

**`AnalysisEngineEarlyPassTest`** (new file at `src/test/kotlin/com/ghostdebugger/analysis/AnalysisEngineEarlyPassTest.kt` — sibling of existing engine tests, not under `/analyzers/`; extends `BasePlatformTestCase` because it exercises real PSI):
- Context with a broken Kotlin file + a clean TSX file that contains a null-safety pattern.
- Assert: result contains both `AEG-SYNTAX-001` (on broken file) and `AEG-NULL-001` (on TSX file).
- Assert: result contains NO non-syntax, non-compile issues on the broken Kotlin file (proves skip mechanism).
- Assert: path normalization handles both `/` and `\` separators in `brokenFilePaths` matching.

### 8.4 Coexistence with legacy tests

`FixtureFactory` and mock-based tests stay unchanged. PSI-dependent tests use `BasePlatformTestCase.myFixture.configureByText(...)` to materialize real `VirtualFile` + `PsiFile` pairs. Each new test class gets a one-line comment at the top identifying which framework it uses.

### 8.5 Plugin verification

`./gradlew verifyPlugin` must pass against the existing `pluginVerification` matrix:
- 2023.2.6 (`since-build` floor)
- 2024.1.6
- 2024.3.2.2
- 2025.1

Verifier will flag `@ApiStatus.Internal` use for `DaemonCodeAnalyzerImpl`. Suppress at the call site:

```kotlin
@Suppress("UnstableApiUsage")
private fun harvestFile(...) { ... }
```

with a one-line justification comment noting the public alternative is insufficient (section 6.4).

## 10. Acceptance criteria

V1.1 is ready to ship when:

1. A fresh-install IDE with the V1.1 plugin, opening a Kotlin file containing `val classment = 1`, shows:
   - A red squiggly on the line.
   - An entry in the Aegis Debug tool window with title `Syntax error: …` and rule ID `AEG-SYNTAX-001`.
   - An entry in the IDE Problems panel.
2. A file with a type mismatch (`val x: Int = "s"`) shows a `AEG-COMPILE-001` entry.
3. A file with a syntax error does NOT produce any `AEG-NULL-001` / `AEG-STATE-001` / `AEG-ASYNC-001` / `AEG-CPX-001` issues, even if the source text would have matched one of those regex patterns.
4. `./gradlew clean test verifyPlugin buildPlugin` passes end-to-end.
5. Analysis runtime on a 200-file project stays under 30 seconds total (including the new passes).
6. `CHANGELOG.md`, `plugin.xml`, `build.gradle.kts`, and the docs listed in section 9.4 are updated and consistent.
