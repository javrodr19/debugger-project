# Aegis Debug V1.2 — Hardening pass (PSI depth, resilient AI parsing, cross-file reanalysis)

**Date:** 2026-04-19
**Status:** Approved — ready for implementation plan
**Version target:** `1.2.0` (bundled release, not patch-sliced)
**Base branch:** `main`
**Target working branch:** `v1.2-hardening`

---

## 0. Summary

V1.2 addresses five recommendations from an external senior-engineer review of the V1.1.2 codebase. All five land in a single bundled release to close the review cleanly, sequenced by dependency:

1. Introduce a shared resilient AI JSON extractor with strategy-level telemetry (Rec 2).
2. Route the Ollama AI pass through the same `AIAnalyzer` that OpenAI already uses; delete the duplicated per-file fan-out in `AnalysisEngine` (Rec 5).
3. Adopt the new JSON extractor across both AI services via a shared `AiIssueMapper` (Rec 2, post-Rec 5).
4. Add few-shot examples to `detectIssues` and `jointFix` prompts (Rec 4).
5. Deepen PSI integration in `SymbolExtractor` for Kotlin and Java; harden the TS/JS regex path; add a new Kotlin-PSI-backed `KotlinNullSafetyAnalyzer` (Rec 1).
6. Teach `reanalyzeFile` to cascade static-only re-analysis to transitive dependents in the `ProjectGraph`, capped by a new setting (Rec 3).

V1.1.2 shipped with 117 green tests and `verifyPlugin` Compatible across IU 2023.2.6 / 2024.1.6 / 2024.3.2.2 / 2025.1. V1.2 must preserve both bars.

---

## 1. Motivation

The senior-engineer review flagged five technical weaknesses in V1.1.2, each with a corresponding recommendation:

| Rec | Weakness it addresses |
|---|---|
| 1 | Regex fragility in `SymbolExtractor` and `NullSafetyAnalyzer` (multi-line statements, content in strings/comments, no semantic types) |
| 2 | Brittle `"[" + substringAfter("[") + "]"` JSON extraction in both `OllamaService` and `OpenAIService` — fails on conversational preamble, markdown fences, nested brackets in strings |
| 3 | `reanalyzeFile` updates only the changed file; a signature change that affects dependents leaves those dependents showing stale status until a full project refresh |
| 4 | Prompts are one-shot; Ollama models in particular drift off the JSON schema without worked examples |
| 5 | Two implementations of the same per-file fan-out pattern (one in `AnalysisEngine.runOllamaPass`, one in `AIAnalyzer` for OpenAI) — duplicated semaphore setup, divergent progress-label conventions, doubled maintenance surface |

Fixing recs 2, 4, 5 in isolation would be fast but would leave the review's two structural concerns (regex fragility in rec 1, cross-file regression gap in rec 3) unaddressed — and those are the ones that directly degrade *finding quality*, not just implementation hygiene. A bundled V1.2 closes the review in a single reviewable release.

## 2. Scope decisions (locked in)

| # | Decision | Rationale |
|---|---|---|
| Q1 | **Bundle all five recs in V1.2.0** (no patch-slice). | Review closes cleanly in one release. Dependency-ordered build (see §7) means each rec can be committed and reviewed independently even inside one release. |
| Q2 | **PSI for Kotlin + Java; TS/JS keeps hardened regex; add `KotlinNullSafetyAnalyzer`.** | IntelliJ Community does not bundle JS/TS PSI (Ultimate/WebStorm only). A dual-path JS/TS PSI (P3) adds maintenance without benefit for Community users and wasn't requested. `KotlinNullSafetyAnalyzer` expands semantic coverage where PSI is strongest. |
| Q3 | **Transitive dependents, static-pass only, capped.** | Static analysis is the signal that goes stale across files; AI pass is per-file semantic review. | Q4 | **Route Ollama through `AIAnalyzer`; keep static-pass fan-out as-is.** | The review's concern is "two AI implementations of the same pattern" — the static pass parallelizes across *analyzers*, not files, so it's a different axis and forcing it into the same helper would hurt readability. |
| Q5 | **Multi-strategy JSON extractor (direct → fenced → bracket-balanced) with telemetry.** | Model misbehavior is a small predictable set of shapes; each strategy addresses one. Per-strategy counters give operator-grade data for future prompt tuning without a user-visible feature. |
| Q6 | **Few-shot on the two JSON-producing prompts only (`detectIssues`, `jointFix`).** | Prose prompts don't benefit from few-shot. Negative examples (F3) backfire on small local Ollama models. |

## 3. Architecture

### 3.1 New file: `ai/AiJsonExtractor.kt`

```kotlin
object AiJsonExtractor {
    sealed interface Result {
        data class Ok(val element: JsonElement, val strategy: Strategy) : Result
        data object Empty : Result
    }
    enum class Strategy { DIRECT, FENCED, BALANCED }

    fun extract(raw: String): Result
    fun telemetrySnapshot(): Map<Strategy, Long>
}
```

- **DIRECT:** try `Json.parseToJsonElement(raw.trim())` first.
- **FENCED:** extract the first `` ```json ... ``` `` (or plain `` ``` ... ``` ``) block and parse it.
- **BALANCED:** scan the raw string for the first `{` or `[`, then walk forward with a state machine that tracks string-literal state (`"..."`, escapes), increments on `{`/`[`, decrements on `}`/`]`; extract the matched span and parse.
- On any success, increment an internal per-strategy counter (`AtomicLong`). On all-strategy failure, return `Empty`.
- Purely a parser; no issue-construction logic here.

### 3.2 New file: `ai/AiIssueMapper.kt`

```kotlin
object AiIssueMapper {
    fun mapIssues(element: JsonElement, filePath: String, fileContent: String): List<Issue>
}
```

Unified issue-construction replacing the duplicated blocks in `OllamaService.parseDetectIssuesResponse` and `OpenAIService.detectIssues`. Handles both `JsonArray` and `JsonObject{issues: [...]}` shapes, skips items missing `type` or `line`, matches existing `IssueType.valueOf` / `IssueSeverity.valueOf` defaulting.

### 3.3 Changes to `AIAnalyzer`

- Add constructor param `labelPrefix: String = "AI: "` for progress-indicator text.
- Expose `DEFAULT_CONCURRENCY_LOCAL = 4` and `DEFAULT_CONCURRENCY_CLOUD = 3` as companion constants.
- Behavior otherwise unchanged — still semaphore-limited per-file fan-out.

### 3.4 Changes to `AnalysisEngine`

- `runOllamaPass` is rewritten to construct `OllamaService`, then `AIAnalyzer(service, progress, concurrency=4, labelPrefix="Ollama: ").analyze(aiContext)`. The inline `Semaphore` + `coroutineScope` + `async` loop is deleted.
- `runOpenAiPass` unchanged structurally; passes `labelPrefix="OpenAI: "` through `aiPassRunner`.
- `OLLAMA_CONCURRENCY` companion constant removed (moves to `AIAnalyzer.DEFAULT_CONCURRENCY_LOCAL`).
- **New public method** `suspend fun analyzeStaticOnly(ctx, indicator): AnalysisResult` — runs the early + late static passes and skips `runAiPass`. Extracts shared private `doStaticPasses(ctx, indicator): StaticPassResult` that encapsulates today's `runStaticPass(earlyAnalyzers)` → broken-file filter → `runStaticPass(lateAnalyzers)` and returns both the early + late issue lists and the filtered context. The public `analyze()` now calls `doStaticPasses`, then `runAiPass` on the filtered context, then merges; `analyzeStaticOnly()` calls `doStaticPasses` and merges without an AI pass. Metrics / hotspots / risks computation stays in the public methods, not in `doStaticPasses`.

### 3.5 Service-level adoption of `AiJsonExtractor` + `AiIssueMapper`

- `OllamaService.parseDetectIssuesResponse` → delete body; `OllamaService.detectIssues` calls `callOllama(...)` → `AiJsonExtractor.extract(raw)` → `AiIssueMapper.mapIssues(element, path, content)` (returning `emptyList()` on `Empty`).
- `OpenAIService.detectIssues` → same structure: HTTP call, then `AiJsonExtractor.extract(rawResponse)` → `AiIssueMapper.mapIssues(...)`. The inline `json.parseToJsonElement(jsonString)` block and the single-issue-JsonObject special case are removed (the mapper handles both shapes).
- `parseFixResponse` in both services is **unchanged** — different shape (code fence + prose), out of scope for V1.2.

### 3.6 New file: `ai/prompts/PromptExamples.kt`

```kotlin
object PromptExamples {
    val DETECT_ISSUES_EXAMPLES: String   // 3 worked examples
    val JOINT_FIX_EXAMPLES: String       // 2 worked examples
}
```

Examples:
- **`DETECT_ISSUES_EXAMPLES`:** (a) null-safety finding (`useState(null)` accessed without guard), (b) circular dep (`type: CIRCULAR_DEPENDENCY`), (c) clean file → `[]`. Formatted as `Input: <code>\nOutput: <json>` blocks so models see the input→output mapping.
- **`JOINT_FIX_EXAMPLES`:** (a) single-file trivial rename, (b) two-file signature change (shows the `fixes: [{filePath, fixedCode}, ...]` shape with multiple entries).

`PromptTemplates.detectIssues` splices `DETECT_ISSUES_EXAMPLES` above the schema block. `PromptTemplates.jointFix` splices `JOINT_FIX_EXAMPLES` similarly. No runtime branching; examples are always included (token budget delta measured in `AnalysisEnginePerfSmoke`).

### 3.7 Parser restructure

```
parser/
  SymbolExtractor.kt              (dispatcher, ~20 lines)
  KotlinPsiSymbolExtractor.kt     (new)
  JavaPsiSymbolExtractor.kt       (new)
  TsJsRegexSymbolExtractor.kt     (new — current TS/JS regex, hardened)
```

- `SymbolExtractor` constructor gains `project: Project?` (nullable — test fixtures and non-PSI callers pass `null`). The dispatcher delegates on `parsedFile.extension`.
- **`KotlinPsiSymbolExtractor`** — given a `Project`, uses `PsiManager.getInstance(project).findFile(vFile)` to obtain a `KtFile`; without a `Project`, uses `PsiFileFactory.getInstance(defaultProject).createFileFromText(name, KotlinLanguage.INSTANCE, content)`. Walks `KtFile.importDirectives`, `KtFile.declarations` for `KtNamedFunction` (captures `hasModifier(KtTokens.SUSPEND_KEYWORD)` as `isAsync`), `KtProperty`, `KtClassOrObject`. On PSI failure, falls back to the retained Kotlin regex (private method `fallbackRegex`).
- **`JavaPsiSymbolExtractor`** — analogous, using `PsiJavaFile`, `PsiImportStatement`, `PsiMethod` (skipping constructors and `isConstructor` / synthetic), `PsiField`, `PsiClass`. Fallback to retained Java regex.
- **`TsJsRegexSymbolExtractor`** — keeps current regexes, adds two hardening steps:
  - **Comment/string masking:** a pre-pass that replaces content inside `'…'`, `"…"`, `` `…` ``, `//…` (to EOL), `/* … */` with space-of-same-length before running regexes. Line numbers preserved.
  - **Multi-line import coalescing:** a second pre-pass that joins lines from an `import {` opener until the matching `}` into a single logical line (again preserving original line numbers for symbol attribution).

### 3.8 New analyzer: `analysis/analyzers/KotlinNullSafetyAnalyzer.kt`

```kotlin
class KotlinNullSafetyAnalyzer : Analyzer {
    override val name = "KotlinNullSafetyAnalyzer"
    override val ruleId = "AEG-NULL-KT-001"
    override val defaultSeverity = IssueSeverity.ERROR
}
```

- Scope: `.kt` files only; single-file (no cross-file type resolution).
- **Scope definition:** "same scope" means the enclosing `KtNamedFunction`, `KtLambdaExpression`, or `KtClassInitializer` that lexically contains the property declaration — **not** cross-function type inference. Access in a different function referring to a `KtProperty` by name is out of scope (the V1 TS/JS `NullSafetyAnalyzer` has the same single-scope limit).
- Algorithm:
  1. Walk the `KtFile`, collect every `KtProperty` whose declared type ends in `?` (nullable) **or** whose initializer is a literal `null`. Record the declaring element and its enclosing scope element.
  2. Walk `KtDotQualifiedExpression` nodes inside each scope; match the receiver by name (via `KtNameReferenceExpression.getReferencedName()`) against collected nullable property names in that same scope. Name-based matching, not full PSI resolution — this is a deliberate simplification for V1.2 (full resolution would need Kotlin's `BindingContext` / `analyzeAndGetBindingContext`, which carries a significant runtime cost and its own failure modes).
  3. A finding is emitted unless the access is guarded by any of: `?.` (safe call — the node would be a `KtSafeQualifiedExpression` instead), `!!` unwrap on the receiver in the same expression, an enclosing `if (x != null) { ... }` / `if (x == null) return`, an enclosing `x?.let { … }` closure, an `x ?: return` / `x ?: throw` immediately before the access, or a reassignment `x = nonNullValue` strictly between declaration and access in the same scope.
- Confidence `0.9` (not `1.0` — no full type inference; we tolerate some miss bias).
- Rule toggleable via existing inspection-profile plumbing.

### 3.9 `GhostDebuggerService.reanalyzeFile` — dependent cascade

- **Call-site change:** today line 636 constructs `val extractor = SymbolExtractor()` with no project. V1.2 changes this (and any other direct construction) to `SymbolExtractor(project)` so the PSI dispatcher has access to `PsiManager`. Tests that construct the extractor in isolation continue to pass `null` and receive the regex path.
- After the existing primary-file update block (lines 615–666 today), add:
  1. `val dependentNodeIds = inMemoryGraph.calculateImpact(nodeId).take(settings.maxDependentsToReanalyze)`.
  2. If empty, done. Otherwise, resolve each dependent node id to a `VirtualFile` via `LocalFileSystem.getInstance().findFileByPath(...)`; run `FileScanner(project).parsedFiles(vFiles)` inside a single `runReadAction`; apply `SymbolExtractor(project).extract(...)` to each resulting `ParsedFile`.
  3. Build one `AnalysisContext` containing all dependents (plus the existing `graph` and `project` fields); call `engine.analyzeStaticOnly(ctx)` (§3.4).
  4. Group returned issues by `filePath`. For each dependent file, compute new status (`ERROR`/`WARNING`/`HEALTHY`), call `inMemoryGraph.updateNode(...)` and push `bridge.sendNodeUpdate(id, status)` + `bridge.sendIssuesForFile(path, issues)` on the Swing dispatcher.
  5. Update the in-memory `currentIssues`: drop old issues whose `filePath` matches any dependent, then add the fresh dependent issues (same pattern the primary-file step already uses).
- New setting: `GhostDebuggerSettings.State.maxDependentsToReanalyze` (default `20`, user-visible in Settings UI). Value `0` disables the dependent cascade entirely.

## 4. Data flow

### 4.1 `reanalyzeFile` with dependents

```
User edits Foo.kt → auto-refresh (7s) → reanalyzeFile("Foo.kt")
  1. PsiDocumentManager.commitAllDocuments()                       (EDT)
  2. FileScanner → ParsedFile(Foo.kt)
  3. SymbolExtractor.extract(Foo.kt)                (dispatcher → Kotlin PSI path)
  4. AnalysisEngine.analyze(ctx=[Foo.kt])           — static + AI on primary
  5. updateIssues(currentIssues − Foo.kt + newFoo)
  6. graph.updateNode(Foo.kt) → bridge.sendNodeUpdate + sendIssuesForFile
  7. dependents = graph.calculateImpact("Foo.kt").take(settings.maxDependentsToReanalyze)
  8. if dependents.isNotEmpty():
       FileScanner → ParsedFile list
       SymbolExtractor on each
       AnalysisEngine.analyzeStaticOnly(ctx=dependents)   (parallel, Semaphore(4))
       for each dependent:
         updateIssues(...); graph.updateNode(...); bridge.sendNodeUpdate + sendIssuesForFile
  9. Done. AI pass never fires for dependents.
```

### 4.2 AI pass (unified, post-Rec 5 + Rec 2)

```
AnalysisEngine.runAiPass()
      │
      ▼ (OLLAMA branch)                               (OPENAI branch)
  OllamaService(...)                                 OpenAIService(...)
                   \                                 /
                    AIAnalyzer(service, progress, concurrency, labelPrefix)
                                       │
                                       ▼
                     for each file (under Semaphore(concurrency)):
                       service.detectIssues(file.path, file.content)
                                       │
                                       ▼
                             (inside each service)
                     raw = HTTP → model response
                     element = AiJsonExtractor.extract(raw)
                                       │
                       ┌───────────────┼───────────────┐
                       ▼               ▼               ▼
                   Ok(DIRECT)    Ok(FENCED)    Ok(BALANCED)     Empty
                       └───────────────┴───────────────┘          │
                                       │                          ▼
                                       ▼                   log.warn + return []
                     AiIssueMapper.mapIssues(element, path, content)
                                       │
                                       ▼
                             returns List<Issue>
```

Telemetry: `AiJsonExtractor` per-strategy counters logged at `GhostDebuggerService.dispose()` at INFO (e.g. `AiJson: DIRECT=412 FENCED=87 BALANCED=31`). No user-facing UI.

## 5. Error handling

**Invariant:** no new `catch (Throwable)` in V1.2. V1.1.2 postmortem established that `catch (Throwable)` in `CompilationErrorAnalyzer.harvestFile` masked the missing `HighlightingSession` setup for months. All new catches are typed `Exception` with an explicit re-throw check for `ProcessCanceledException` (per IntelliJ convention).

| Boundary | Success path | Failure path | Log level |
|---|---|---|---|
| `AiJsonExtractor.extract` | `Result.Ok(element, strategy)` | `Result.Empty` if all strategies fail | `log.warn("AiJson: all strategies failed, len=${raw.length}")` at service caller |
| `AIAnalyzer.analyze` per-file | List of issues | Per-file exception → `emptyList()` for that file; siblings unaffected | `log.warn("AI pass failed for $path", e)` |
| `AIAnalyzer.analyze` whole pass | Aggregated list | Top-level exception → bubbled to `AnalysisEngine.runAiPass`, handled as `FALLBACK_TO_STATIC` | Existing behavior preserved |
| `KotlinPsiSymbolExtractor.extract` | `ParsedFile` with populated symbols | PSI throws or file not a `KtFile` → private regex fallback | `log.warn("Kotlin PSI extract failed for $path, falling back to regex", e)` |
| `JavaPsiSymbolExtractor.extract` | Same | Same — regex fallback | Same |
| `KotlinNullSafetyAnalyzer.analyze` | Issue list | Per-file exception → `emptyList()` for that file, `runOne` semantics preserved | `log.warn("KotlinNullSafetyAnalyzer failed for $path", e)` |
| `reanalyzeFile` primary-file step | Issues + graph updates for `filePath` | Any exception → fall through to full `analyzeProject()` | `log.warn("Targeted re-analysis failed for $filePath", e)` |
| `reanalyzeFile` dependents step | Bridge updates per dependent | Per-dependent exception logged, loop continues; primary-file update still committed | `log.warn("Dependent re-analysis failed for $depPath", e)` |
| Any coroutine body | Progress check | `ProcessCanceledException` re-thrown unconditionally from all new catches | — |

## 8. Out of scope

- **JS/TS PSI** — Community IDE lacks the plugin; dual-path complexity unjustified.
- **Negative few-shot examples** — known to backfire on small local Ollama models.
- **User-visible concurrency settings** — defer to later if users ask.
- **Auto-refresh timer tuning** (the 7s delay) — review noted it but did not recommend a change.
- **`parseFixResponse` unification** — different shape (code fence + prose) from `detectIssues`; worth its own pass when `suggestFix` also gets few-shot treatment. Not V1.2.
- **Runtime-confirmed provenance, streaming AI, Python support** — all V2 per `aegis_debug_roadmap_v2_to_v5.md`.

## 9. Risks and mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| PSI migration slows analysis on large repos (PSI is heavier than regex) | Medium | `AnalysisEnginePerfSmoke` is the budget guard. If it regresses >20%, gate the PSI path behind a project-level setting and investigate the specific hotspot. |
| `KotlinNullSafetyAnalyzer` false positives in DI / late-init codebases | Medium–High | Confidence `0.9`, single-file scope, conservative-miss bias; rule toggleable. Test set explicitly covers shadowed-variable and reassigned-before-access cases. |
| `AiJsonExtractor` bracket-balance strategy over-parses garbage into phantom issues | Medium | `AiIssueMapper` is strict — missing `type` or `line` → skip. Balanced scanner is string-literal-aware (tested). On pathological input, `Empty` + log is better than "best effort." |
| Concurrency refactor changes observable timing on Ollama pass | Low | Ollama already ran under `Semaphore(4)`; `AIAnalyzer` preserves that. `AnalysisEngineParallelStaticPassTest` and `OllamaServiceParseTest` guard the behavior. |
| Transitive dependents cascade on hub files (`types.ts` → 50-file rescan) | Medium | `maxDependentsToReanalyze=20` default; `calculateImpact` result is `take()`'d. Kill switch at `0`. Static-only path keeps cost bounded even at the cap. |
| PSI extractors fail on very broken files and regress symbol coverage | Low | Regex fallback retained in every PSI extractor, exercised by a "malformed input" test in each. |

## 10. Rollback

Each rec is a self-contained commit (commit list in §7). Post-merge regression on any single rec → `git revert <sha>` is clean thanks to dependency ordering — reverting Rec 3 or Rec 1 doesn't require unwinding Recs 2/4/5. The bundled *release* is V1.2.0; any single-rec regression ships as V1.2.1.

## 11. Acceptance criteria

V1.2.0 ships when all of the following hold on `v1.2-hardening` at HEAD:

1. All 117 V1.1.2 tests still pass.
2. ~44 new V1.2 tests exist and pass (total ~161).
3. `verifyPlugin` reports Compatible on IU 2023.2.6, 2024.1.6, 2024.3.2.2, 2025.1 with zero compatibility problems.
4. `AnalysisEnginePerfSmoke` stays within its existing budget (±20% tolerance).
5. `AnalysisEngine.runOllamaPass` no longer contains an inline `Semaphore` — all AI per-file fan-out lives in `AIAnalyzer`.
6. `OllamaService.parseDetectIssuesResponse` is deleted; `OpenAIService.detectIssues` no longer contains the inline `substringAfter("[")` logic; both services route JSON parsing through `AiJsonExtractor` → `AiIssueMapper`.
7. `PromptTemplates.detectIssues` and `jointFix` render prompts that include the fixed example bodies verbatim (assert in tests).
8. `SymbolExtractor` dispatches to PSI extractors for `.kt` and `.java`; TS/JS goes through the hardened regex path.
9. `KotlinNullSafetyAnalyzer` is wired into `AnalysisEngine`'s default analyzer list.
10. Editing a file with dependents triggers `sendNodeUpdate` + `sendIssuesForFile` for those dependents (capped), without invoking the AI pass on them.
