# Aegis Debug V1.1.2 — Test infrastructure: enable `BasePlatformTestCase` suite

**Date:** 2026-04-18
**Status:** Superseded by implementation — see postmortem below
**Version target:** `1.1.2` (patch on V1.1.1)
**Base branch:** `main`

---

## Postmortem (retained for archival)

> The central hypothesis of this spec — that upgrading `org.jetbrains.intellij.platform` `2.2.1` → `2.14.0` resolves the `IndexingTestUtil.waitUntilIndexesAreReady` hang — **proved wrong** during implementation. The hang was a classpath collision: stock `kotlinx-coroutines-core:1.9.0` (pulled in directly and transitively via `mockk`) shadowed the IntelliJ-forked `kotlinx-coroutines-core-jvm-*-intellij.jar`, causing `UnindexedFilesScanner` to crash with `NoSuchMethodError` inside an unobservable coroutine, which left the indexing barrier un-signaled forever.
>
> The real fix: `kotlinx-coroutines-core` as `compileOnly` + a configuration-level `exclude` on `runtimeClasspath` and `testRuntimeClasspath`. The 2.14.0 plugin bump was retained because its `TestFrameworkType` wiring is cleaner, not because it was load-bearing for the hang.
>
> Implementation also uncovered and fixed a latent production bug in `CompilationErrorAnalyzer` (missing `DaemonProgressIndicator` / `HighlightingSession` setup around `runMainPasses`, silently swallowed by `catch (Throwable)`). See `CHANGELOG.md` §1.1.2 for the shipped scope. This spec is retained unedited below as a record of the original (incorrect) investigation.

---

## 0. Summary

V1.1 introduced three `BasePlatformTestCase` tests (`PsiSyntaxAnalyzerTest`, `CompilationErrorAnalyzerTest`, `AnalysisEngineEarlyPassTest`). V1.1.1 added two more (`FileScannerDocumentReadTest`, `AnalysisEnginePostEditRerunTest`). None of them have ever actually executed in this repository: first silently skipped (no `junit-vintage-engine` on the classpath, so JUnit Platform couldn't discover JUnit 3 `TestCase` descendants), then — once V1.1.1 added the vintage engine — hung indefinitely inside the IntelliJ test fixture's `setUp()` while waiting on indexing.

V1.1.2 is an infrastructure-only patch. No production code changes. The single hypothesis to test: upgrading `org.jetbrains.intellij.platform` from `2.2.1` to `2.14.0` resolves the sandbox-indexing hang (their 2.3+ changelog reworked `prepareTestSandbox` / index priming). Success criterion is that all five `BasePlatformTestCase` tests start producing green `TEST-*.xml` reports, which retrospectively validates the V1.1.1 fix.

---

## 1. Root cause (evidence)

Two stacked defects, each independently sufficient to produce "green build, zero tests":

### 1a. Missing vintage engine (fixed post-V1.1.1, carried into V1.1.2)

`build.gradle.kts` uses `useJUnitPlatform()` with only `junit-jupiter-engine:5.11.4` on the test classpath. `BasePlatformTestCase` extends JUnit 3 `TestCase`, so JUnit Platform cannot discover it without `junit-vintage-engine`. Without vintage: test runner reports "100% pass" after discovering 0 eligible tests from the five classes.

Fix already applied during V1.1.1 review:

```kotlin
// build.gradle.kts
testRuntimeOnly("org.junit.vintage:junit-vintage-engine:5.11.4")
```

### 1b. Fixture indexing hang (exposed when 1a was fixed)

Thread dump captured from the V1.1.1 verification run (≈30 min elapsed, no progress):

```
"AWT-EventQueue-0"  — TIMED_WAITING (sleeping)
  java.lang.Thread.sleep
  at PlatformTestUtil.waitWithEventsDispatching:1244
  at IndexingTestUtil.waitUntilIndexesAreReady
  at CodeInsightTestFixtureImpl.ensureIndexesUpToDate:338
  at CodeInsightTestFixtureImpl.setUp:1380
  at BasePlatformTestCase.setUp:44

"Test worker" — TIMED_WAITING (parking)
  com.intellij.util.concurrency.Semaphore.waitFor
  at LaterInvocator.invokeAndWait
  at EdtTestUtil.runInEdtAndWait  (waiting on EDT to finish setUp)
```

Tests eventually hit `java.lang.AssertionError at PlatformTestUtil.java:1234` — the index-wait timeout assertion. A targeted single-test run (`--tests FileScannerDocumentReadTest.testParsedFilesFallsBackToVfsWhenNoDocument`) hit the 180s wall-clock timeout without producing an XML report, confirming the hang is per-test fixture `setUp`, not load.

Build log warns:

```
[org.jetbrains.intellij.platform] IntelliJ Platform Gradle Plugin is outdated: 2.2.1.
Update `org.jetbrains.intellij.platform` to: 2.14.0
```

Version `2.2.1` predates the `prepareTestSandbox` / index-priming rework that landed around `2.3.0` – `2.5.0`. The hang is consistent with a sandbox that bootstraps without priming the index state the `CodeInsightTestFixture` expects.

## 2. Scope decisions (locked in)

| # | Decision | Rationale |
|---|---|---|
| Q1 | **Upgrade `org.jetbrains.intellij.platform` from `2.2.1` → `2.14.0`** | Targeted fix. The plugin's test-sandbox bootstrapping was reworked between 2.2 and 2.14 (release notes mention index priming). Likely resolves the hang with zero test-code churn. |
| Q2 | **Keep `junit-vintage-engine:5.11.4` on `testRuntimeOnly`** | Required to discover JUnit 3 `TestCase` descendants under `useJUnitPlatform()`. Removing it regresses to silent skip. |
| Q3 | **Do not rewrite tests to `LightPlatformTestCase` unless Q1 fails** | `BasePlatformTestCase` is the correct abstraction for tests that need a real PSI tree on a real `VirtualFile` + `Document` — exactly the surface V1.1.1 fixed. `LightPlatformTestCase` loses the Document subsystem we're testing. Reserve as contingency. |
| Q4 | **Do not bump the target IDE (`intellijIdeaCommunity("2024.3.2")`) in this patch** | Orthogonal risk. If Q1 does not resolve the hang, an IDE bump is a V1.1.3 concern, not bundled here. |
| Q5 | **No production code changes** | V1.1.2 is infrastructure-only. `FileScanner`, `AnalysisEngine`, and all analyzers stay untouched. Version bump is a patch (`1.1.1` → `1.1.2`). |
| Q6 | **No new tests** | Success is measured by previously-silent tests now executing and passing. The five tests already exist. |

### Non-goals

- Not migrating the suite to JUnit 5 Jupiter.
- Not introducing a separate `integrationTest` source set.
- Not touching the `pluginVerification` IDE matrix (2023.2.6, 2024.1.6, 2024.3.2.2, 2025.1).
- Not changing CI workflow files.
- Not changing webview tasks (`npmInstallWebview`, `buildWebview`) — they use `Exec`, independent of the platform plugin.

## 5. Risks from the gradle-plugin upgrade

| Risk | Detection | Mitigation |
|---|---|---|
| API change breaks `intellijPlatform { pluginConfiguration { ... } }` DSL | Build fails at `./gradlew clean build` | 2.x line is backwards-compatible with deprecation warnings through 2.14. Follow migration notes if any DSL key was renamed. |

| `since-build = "232.0"` / `until-build = "261.*"` range validation changes in 2.14 | `patchPluginXml` emits a warning | Adjust the range if the new plugin warns it's out of bounds. |
| `instrumentCode.enabled = false` override at line 144 renamed | `./gradlew jar` fails or re-enables instrumentation | Spot-check after upgrade. Fallback is to replicate via the new API shape. |
| `testFramework(TestFrameworkType.Platform)` constant removed / renamed | Compile failure in `build.gradle.kts` | Check 2.x API docs; name may have changed to `TestFrameworkType.Platform` (unchanged) or a new constant. |
| Hang persists despite plugin upgrade | Test run hangs or times out as today | Enter Phase C (§6.3) diagnostics. |

## 6. Implementation sequencing

### 6.1 Phase A — upgrade gradle plugin (minimum viable fix)

1. Branch `v1.1.2-test-infra` off `main`.
2. Edit `build.gradle.kts` line 8: `version "2.2.1"` → `version "2.14.0"`.
3. `./gradlew --stop` to clear the daemon.
4. `./gradlew clean build --refresh-dependencies` — fixes any DSL-level rename eagerly.
5. If the build is green, continue to Phase B. If it fails at DSL level, triage per §5, fix in-place, then continue.

### 6.2 Phase B — validate the test suite

1. `./gradlew test`.
2. Confirm the run terminates in minutes, not hours.
3. Inspect `build/test-results/test/` — expect `TEST-*.xml` for all five previously-silent classes plus the existing 40.
4. Inspect each of the five new XMLs: `failures=0 errors=0`.
5. `./gradlew verifyPlugin` across the existing IDE matrix — confirm no new verifier warnings introduced by 2.14.
6. `./gradlew buildPlugin` — confirm the deployable ZIP is produced.

### 6.3 Phase C — contingency (only if Phase B's hang persists)

1. Capture fresh `jstack` during the hang. Compare to the V1.1.1 dump to see whether the stack moved.
2. Add diagnostic system properties to `tasks.test`:

   ```kotlin
   tasks.test {
       useJUnitPlatform()
       systemProperty("idea.force.use.core.classloader", "true")
       systemProperty("idea.use.native.fs.for.win", "false")
       jvmArgs("-Didea.home.path=${layout.buildDirectory.dir("idea-sandbox").get().asFile.absolutePath}")
   }
   ```

3. Re-run and capture a third `jstack`.
4. Narrow to a single test class: `./gradlew test --tests PsiSyntaxAnalyzerTest` to isolate whether the hang is universal or test-specific.
5. If narrowed, inspect that test's `myFixture.configureByText` usage — particular extension types trigger particular index extensions.
6. Clean gradle caches as a local-contamination ruleout:

   ```bash
   rm -rf ~/.gradle/caches/modules-2/files-2.1/com.jetbrains.intellij*
   rm -rf ~/.gradle/caches/modules-2/files-2.1/org.jetbrains.intellij*
   ```

### 6.4 Phase D — last-resort fallback (only if Phases B and C fail)

Migrate the five tests to `LightJavaCodeInsightFixtureTestCase`. This loses coverage for the Document↔VFS divergence that V1.1.1 fixes, so pair the migration with a manual acceptance checklist (V1.1.1 spec §9) that must be walked before every release. This is a degraded state, not a target.

### 6.5 Wrap-up (after A/B succeed)

1. Bump `build.gradle.kts` line 12: `version = "1.1.1"` → `"1.1.2"`.
2. Bump `plugin.xml` line 4.
3. Prepend `CHANGELOG.md` entry (§7).
4. Append V1.1.2 amendment to `docs/aegis_v1_history.md`.
5. `./gradlew clean test verifyPlugin buildPlugin` end-to-end.
6. Open PR.

## 9. Blind spots and known risks to the plan

- **Hypothesis risk (primary).** The plan rests on the belief that upgrading `org.jetbrains.intellij.platform` from 2.2.1 to 2.14.0 fixes the indexing hang. Evidence is circumstantial: the build log flags the plugin as outdated, and the 2.3+ changelogs describe sandbox-priming rework. If the real root cause is elsewhere (JDK 21 + 2024.3.2 fixture interaction, a missing system property, a host-specific VFS quirk), Phase A will not resolve it and we fall to Phase C.
- **Verifier matrix risk.** Upgrading the gradle plugin can surface plugin-verifier issues against older IDE targets (especially 2023.2.6). Budget one triage cycle; the `until-build = "261.*"` range may need tightening.
- **Local-contamination risk.** The hang may be partly caused by stale entries in `~/.gradle/caches/modules-2/files-2.1/com.jetbrains.intellij*`. A clean-cache rebuild should be part of Phase B validation before concluding the upgrade succeeded.
- **Duration risk.** Once the five `BasePlatformTestCase` tests actually run, total `./gradlew test` time will grow from ≈30s to several minutes on cold starts, because each class boots a real IntelliJ. Not a blocker, but worth noting for CI once CI exists.
