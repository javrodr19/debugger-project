# Aegis Debug V1.1.1 — Stale-content bug fix (re-analysis sees saved, not current, editor text)

**Date:** 2026-04-17
**Status:** Design approved, ready for implementation
**Version target:** `1.1.1` (patch on V1.1.0)
**Base branch:** `main`

---

## 0. Summary

V1.1 introduced two early-pass PSI-based analyzers (`AEG-SYNTAX-001`, `AEG-COMPILE-001`). They read file text through the IntelliJ `Document`/PSI tree, which always reflects the live editor buffer. Every other analyzer — `NullSafetyAnalyzer`, `StateInitAnalyzer`, `AsyncFlowAnalyzer`, `CircularDependencyAnalyzer`, `ComplexityAnalyzer`, and the `AIAnalyzer` wrapper around Ollama / OpenAI — operates on `ParsedFile.content`, which is loaded by `FileScanner` via `VirtualFile.contentsToByteArray()`. That call returns last-saved disk bytes, not the editor buffer.

Observed consequence: a user who corrects an `AEG-NULL-001`, `AEG-STATE-001`, `AEG-ASYNC-001`, `AEG-CPX-001`, or AI-sourced issue and re-runs analysis without first saving the file still sees the issue flagged. The same flow on an `AEG-SYNTAX-001` or `AEG-COMPILE-001` issue clears correctly — hence the "only the other errors" framing.

V1.1.1 unifies the text source across all analyzers by having `FileScanner` prefer the live IDE `Document` text, falling back to `VirtualFile.contentsToByteArray()` only when no document exists. No other surface changes.

## 1. Root cause (evidence)

Single instrumented read site:

```
src/main/kotlin/com/ghostdebugger/parser/FileScanner.kt:67
    val content = String(vf.contentsToByteArray(), Charsets.UTF_8)
```

Every downstream consumer of `ParsedFile.content` and `ParsedFile.lines` (lazy on `content`) inherits whatever `FileScanner` produced:

| Consumer | File | Uses |
|---|---|---|
| `NullSafetyAnalyzer` | `analyzers/NullSafetyAnalyzer.kt:19` | `file.lines` |
| `StateInitAnalyzer` | `analyzers/StateInitAnalyzer.kt:19` | `file.lines` |
| `AsyncFlowAnalyzer` | `analyzers/AsyncFlowAnalyzer.kt:19` | `file.lines` |
| `ComplexityAnalyzer` | `analyzers/ComplexityAnalyzer.kt:30` | `file.lines` |
| `AIAnalyzer` | `analyzers/AIAnalyzer.kt:38` | `file.content` (sent to provider) |
| `AnalysisEngine.runOllamaPass` | `AnalysisEngine.kt:253` | `file.content` (sent to Ollama) |
| `SymbolExtractor` | `parser/SymbolExtractor.kt` | `file.content` (imports feed the graph that `CircularDependencyAnalyzer` reads) |

Contrast with the early analyzers:

| Analyzer | File | Text source |
|---|---|---|
| `PsiSyntaxAnalyzer` | `analyzers/PsiSyntaxAnalyzer.kt:27-30` | `PsiManager.findFile(virtualFile)` → live PSI tree |
| `CompilationErrorAnalyzer` | `analyzers/CompilationErrorAnalyzer.kt:52` | `FileDocumentManager.getDocument(virtualFile)` → live Document |

`VirtualFile.contentsToByteArray()` is documented to return stale bytes when a document is unsaved. IntelliJ does not force-save on every action, so any flow where the user edits in the editor and triggers "Analyze Project" before the editor has been saved (Ctrl+S, autosave on focus loss, compile/run, etc.) hits this bug.

Secondary evidence from the file lifecycle:

- `AnalysisEngine.analyze()` at `AnalysisEngine.kt:71-75` clears `file.content = ""` after the run to reclaim RAM; this is benign because each call to `GhostDebuggerService.performAnalysis()` constructs fresh `ParsedFile` instances (`GhostDebuggerService.kt:342-344`, `reanalyzeFile` at `:624-626`). State does not leak across analyses. The bug is purely at the read site.
- `FixApplicator.FixWriter.Default` (`fix/FixApplicator.kt:62`) explicitly calls `fdm.saveDocument(document)` after a plugin-applied fix, which is why plugin-driven "Apply Fix" flows do not trigger this bug — the disk bytes are fresh by the time `reanalyzeFile` runs. The bug is scoped to **manual user edits followed by re-analysis**.

## 2. Scope decisions (locked in)

| # | Decision | Rationale |
|---|---|---|
| Q1 | **Prefer live Document text in `FileScanner.parsedFiles()`; fall back to `VirtualFile.contentsToByteArray()` when no Document exists** | Matches the PSI-based early analyzers' semantics. Non-invasive: does not force-save the user's buffer, does not touch VFS refresh machinery. Single read site, single change. |
| Q2 | **Commit PSI documents once per analysis run, before the early pass** | Belt-and-suspenders: `PsiDocumentManager.commitAllDocuments()` guarantees the PSI tree is in sync with the Document for PSI-based analyzers, avoiding a narrow race where a very recent keystroke has updated the Document but not yet PSI. Cheap (~milliseconds), runs on EDT via `invokeAndWait`. |
| Q3 | **No autosave** | Rejected alternative: call `FileDocumentManager.saveAllDocuments()` at analysis start. Writes the user's buffer to disk without consent. Unnecessary once Q1 is in place because the in-memory text path is sufficient. |
| Q4 | **No change to `ParsedFile` shape or `AnalysisContext` API** | `content: String` stays mutable, `lines` stays lazy on `content`. The change is isolated to how `content` is populated. Downstream analyzers and the AI pass keep working unchanged. |
| Q5 | **No new issue types, no new rule IDs, no new analyzers** | This is a correctness fix on existing analyzers, not a detection change. Version bump is a patch (`1.1.0` → `1.1.1`), not minor. |

### Non-goals

- Not changing the VFS refresh strategy.
- Not introducing document-change listeners or incremental analysis.
- Not adding a user-facing "unsaved changes" warning.
- Not touching `AnalysisEngine` phase ordering, broken-file skip, or the merge logic.
- Not reworking `FixApplicator`; plugin-applied fixes already save explicitly.

### 3.2 Change in `FileScanner.parsedFiles()`

Current (`FileScanner.kt:64-79`):

```kotlin
fun parsedFiles(virtualFiles: List<VirtualFile>): List<ParsedFile> {
    return virtualFiles.mapNotNull { vf ->
        try {
            val content = String(vf.contentsToByteArray(), Charsets.UTF_8)
            ParsedFile(virtualFile = vf, path = vf.path, extension = vf.extension ?: "", content = content)
        } catch (e: Exception) {
            log.warn("Could not read file: ${vf.path}", e)
            null
        }
    }
}
```

New:

```kotlin
fun parsedFiles(virtualFiles: List<VirtualFile>): List<ParsedFile> {
    val fdm = FileDocumentManager.getInstance()
    return virtualFiles.mapNotNull { vf ->
        try {
            val document = fdm.getCachedDocument(vf) ?: fdm.getDocument(vf)
            val content = document?.text
                ?: String(vf.contentsToByteArray(), Charsets.UTF_8)
            ParsedFile(virtualFile = vf, path = vf.path, extension = vf.extension ?: "", content = content)
        } catch (e: Exception) {
            log.warn("Could not read file: ${vf.path}", e)
            null
        }
    }
}
```

Notes:

- `getCachedDocument` returns the Document only if it is already loaded (editor is open or another subsystem already materialised it). Cheap, no I/O.
- `getDocument` is the fallback: it materialises a Document, which for a non-editor-open file is equivalent to loading the bytes from disk. On very large files or binary files it can return null; we fall back to `contentsToByteArray()` in that case.
- Both calls require a read lock. Both existing call sites of `FileScanner.parsedFiles` are already wrapped in `ApplicationManager.getApplication().runReadAction { … }` (`GhostDebuggerService.kt:342-344` for the full scan, `:624-626` for `reanalyzeFile`). No new read-action wrapping needed.
- The lazy `ParsedFile.lines` still works unchanged: it derives from whichever `content` string we stored.

### 3.3 One-line commit in `AnalysisEngine.analyze()`

Inserted at the very top of `AnalysisEngine.analyze(context, indicator)`, before the first progress message:

```kotlin
ApplicationManager.getApplication().invokeAndWait {
    PsiDocumentManager.getInstance(context.project).commitAllDocuments()
}
```

Why `invokeAndWait`: `commitAllDocuments` must run on the EDT with a write-intent lock. `analyze` is called from a background coroutine (`GhostDebuggerService.performAnalysis` launched via `ProgressManager.run`). Cost is milliseconds on a committed-clean project, tens of milliseconds when a recent keystroke is pending. Documented ordering guarantee afterward: PSI mirrors Document for all analyzers that dereference PSI (`PsiSyntaxAnalyzer`, `CompilationErrorAnalyzer`, and any future PSI-based analyzer). This is belt-and-suspenders for Q2 and does not affect late analyzers, which operate on `ParsedFile.content` (now Document-backed after section 3.2).

## 4. Behavior after the fix

| Scenario | V1.1.0 (current) | V1.1.1 (after fix) |
|---|---|---|
| User edits a file to fix `AEG-NULL-001`, **saves**, re-analyses | Clears ✓ | Clears ✓ |
| User edits a file to fix `AEG-NULL-001`, **does not save**, re-analyses | Still flagged ✗ | Clears ✓ |
| User edits a file to fix `AEG-SYNTAX-001`, does not save, re-analyses | Clears ✓ | Clears ✓ (unchanged) |
| User edits a file to fix `AEG-COMPILE-001`, does not save, re-analyses | Clears ✓ | Clears ✓ (unchanged) |
| Plugin-applied fix via "Apply Fix" button | Clears ✓ (save is explicit in `FixApplicator`) | Clears ✓ (unchanged) |
| Unopened file with no cached Document | Reads from VFS bytes ✓ | Reads from VFS bytes via fallback ✓ |
| Binary file / `getDocument` returns null | Reads from VFS bytes ✓ | Falls back to VFS bytes ✓ |
| Very large file (> Document load threshold) | Reads from VFS bytes ✓ | Falls back to VFS bytes ✓ |

The only behavior difference is the unsaved-edit path, which is the bug row.

## 5. Edge cases

| Case | Behavior |
|---|---|
| `FileDocumentManager.getDocument(vf)` returns null (binary, too-large, unsupported) | Fall through to `vf.contentsToByteArray()`. Same as today. |
| `document.text` is empty (file was truncated to empty in editor) | `ParsedFile.content = ""`. `lines` yields `[""]`. All regex-based analyzers emit nothing. PSI pass still sees the live empty file. Correct. |
| Document is loaded but reflects a state different from disk AND the file has been externally modified (editor vs. disk conflict) | We return the editor's Document text. This matches what the user sees in the IDE and what PSI reports, so detections are consistent. Consistent with how syntax/compile errors already behave in V1.1.0. |
| File is open in editor but not in focus; edits pending but not saved | `getCachedDocument` hits; we read live edits. Bug fixed. |
| Two rapid analyses in succession (e.g., user double-clicks "Analyze") | `analysisLock` and `activeAnalysisIndicator?.cancel()` in `GhostDebuggerService.analyzeProject` (`:280-282`) already cancel the prior run. No new concurrency concern introduced. |
| `commitAllDocuments` throws on a broken-state project | Wrap in a best-effort `runCatching`; fall through to analysis. `PsiDocumentManager` rarely throws under normal conditions but we should not fail the whole analysis on a PSI commit hiccup. |

## 6. Threading and IntelliJ Platform correctness

- `FileDocumentManager.getCachedDocument` — safe off-EDT, requires no lock.
- `FileDocumentManager.getDocument` — requires read lock. Both current callers already provide one via `runReadAction`.
- `PsiDocumentManager.commitAllDocuments` — must run on EDT with the write-intent lock. The chosen call site uses `invokeAndWait` from a background coroutine context; this is the IntelliJ-sanctioned idiom.
- No modification to `ParsedFile` → no serialization impact (it's not serialized to JSON; `Issue` and `AnalysisResult` are, both unchanged).

## 7. Testing strategy

### 7.1 Unit / integration

Two new tests under `src/test/kotlin/`, both extending `BasePlatformTestCase` to get real `VirtualFile` + `Document` pairs:

**`parser/FileScannerDocumentReadTest.kt`**
- Creates a Kotlin file with initial content `val x = 1`, configures it via `myFixture.configureByText`.
- Mutates the Document in-place (editor-style edit) to `val x = 2` without saving.
- Calls `FileScanner(project).parsedFiles(listOf(vf))`.
- Asserts `parsedFiles.first().content` equals `val x = 2` (live Document), not `val x = 1` (original VFS bytes).
- Second case: creates a file, never opens it, confirms fallback path loads the disk bytes (the test fixture writes them).

**`analysis/AnalysisEnginePostEditRerunTest.kt`**
- Creates a `.tsx` file with a `NullSafetyAnalyzer`-triggering pattern (`let x = null; x.foo;`).
- Runs `AnalysisEngine.analyze(...)` → expect one `AEG-NULL-001`.
- Mutates the Document (unsaved) to `let x = { foo: 1 }; x.foo;`.
- Re-runs `AnalysisEngine.analyze(...)` on a freshly-scanned context (same flow as `GhostDebuggerService.performAnalysis`).
- Assert: zero `AEG-NULL-001` issues on the second run. This is the regression test that would have caught the bug.
- Parallel case for `StateInitAnalyzer` (`AEG-STATE-001`) to cover multiple late analyzers in one test file.

### 7.2 Contract / regression

- `AnalyzerContractTest` is unchanged (no new analyzers, no new rule IDs).
- `./gradlew clean test verifyPlugin buildPlugin` must pass end-to-end against the existing `pluginVerification` matrix (2023.2.6, 2024.1.6, 2024.3.2.2, 2025.1).
- Manual acceptance: see section 9.

### 7.3 Why these specific tests

The current test suite has no test that exercises the "Document-dirty, VFS-clean" split. That is exactly the state under which the bug manifests. `BasePlatformTestCase.myFixture.configureByText` creates that state cleanly: it populates a `VirtualFile` with initial bytes, loads a `Document`, and then permits in-memory edits without saving. The assertion is simply "what does `FileScanner` return?" and "what does `AnalysisEngine` report?" under that condition.

## 9. Acceptance criteria

V1.1.1 is ready to ship when all of the following hold on a fresh `./gradlew runIde` session:

1. Open a `.tsx` file in the sandbox project with:
   ```tsx
   const [items, setItems] = useState();
   items.map(i => i);
   ```
   Run "Analyze Project". See one `AEG-STATE-001` on the `items.map` line.
2. In the editor, edit the first line to `const [items, setItems] = useState([]);`. **Do not save.** The editor tab shows the modified indicator.
3. Run "Analyze Project" again. The `AEG-STATE-001` entry disappears from the tool window and the IDE Problems panel.
4. Parallel flow with `AEG-NULL-001` (`let x = null; x.foo;` → `let x: { foo: number } = { foo: 1 }; x.foo;`) behaves the same way.
5. A file with a syntax error (`val classment =`) fixed in-editor without saving still clears its `AEG-SYNTAX-001` on re-analysis (regression check on V1.1 behavior).
6. "Apply Fix" on a deterministic fixer-eligible issue still clears the issue from the tool window and subsequent full analyses do not re-emit it (regression check on V1.0 behavior).
7. `./gradlew clean test verifyPlugin buildPlugin` completes with zero failures and no new verifier warnings.
8. `CHANGELOG.md`, `build.gradle.kts`, and `plugin.xml` carry `1.1.1` and are mutually consistent.
